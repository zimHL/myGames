# myGames

给自己孩子做的益智小游戏合集。

## 当前内容

- 首页游戏大厅
- 数独
- 井字棋
- 记忆翻牌

## 当前结构

- `index.html`
  首页，负责展示游戏列表
- `data/games.js`
  首页游戏配置
- `games/sudoku/index.html`
  数独
- `games/tic-tac-toe/index.html`
  井字棋
- `games/memory-match/index.html`
  记忆翻牌

## 后续优化

### 1. 桌面端与移动端是否分开

当前不建议直接拆成两套页面。

原因：

- 当前项目规模还不算大，首页约 60 行，单个游戏页大约 570 到 680 行，仍然处于可维护范围。
- 现在每个游戏都是独立单页，如果马上拆成 `mobile` / `desktop` 两套，会把后续维护成本直接翻倍。
- 这类小游戏的核心逻辑相同，差异主要集中在排版、间距、控件密度、棋盘大小，不值得复制整套逻辑。

建议方案：

- 保持一套 HTML / JS 逻辑
- 通过代码层做响应式适配
- 优先区分“窄屏触控场景”和“宽屏桌面场景”

### 2. 如何区分桌面端和移动端

优先使用代码层区分，不建议在 Nginx 层按设备分发不同页面。

推荐做法：

- 样式层：使用 CSS `@media` 断点控制布局、字号、间距、按钮高度、棋盘尺寸
- 逻辑层：如有必要，使用 `window.matchMedia()` 判断是否进入窄屏模式，再切换少量交互细节
- 组件层：后续如果游戏越来越多，可以抽出统一的“移动端紧凑模式”样式规则

不推荐优先使用 Nginx 区分的原因：

- 依赖 `User-Agent` 判断，容易误判
- 后续缓存、调试、分享链接都会更麻烦
- 项目目前是纯静态小游戏，Nginx 分流带来的收益很小，复杂度却明显增加

### 3. 什么时候再考虑真正分端

当出现下面这些情况时，再考虑拆成桌面版和手机版：

- 桌面端需要额外的信息面板、排行榜、侧边栏
- 移动端需要极致紧凑布局，而桌面端需要完全不同的信息组织
- 某个游戏为了桌面键鼠和移动触控，要维护两套明显不同的交互模型

如果后面真的要分端，优先顺序建议是：

1. 先继续保持同一页面，增加 `desktop` / `compact` 两套样式规则
2. 再把公共逻辑抽出来，避免复制游戏算法
3. 最后如果差异仍然很大，再拆成独立桌面视图和移动视图

### 4. 下一阶段的具体优化方向

- 统一三个游戏在桌面端和移动端的“首屏可见优先”布局策略
- 给每个游戏补一套明确的桌面断点规则，而不是只压移动端
- 抽出统一的页面头部、工具栏、状态条、棋盘容器样式
- 后续新增游戏时，优先复用统一模板，避免每个页面各自演化
